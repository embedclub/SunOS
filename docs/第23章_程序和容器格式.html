23 程序和容器格式

摘要

编译后的网络以两层文件的形式到达引擎：一个调度描述符（dispatch descriptor），即一个 FlatBuffer，用于命名操作并绑定缓冲区；以及一个硬件容器（hardware container）。硬件容器是一个 Mach-O 形状的文件，魔数为 0xbeefface，包含寄存器写入程序和权重库。描述符跟踪的是调度次数而非操作次数：任何深度的融合图都缩减为一个推理操作，前后由输入和输出转换（cast）包围，而桥接切割（bridge cut）则每个段增加一个推理操作。两者之间是硬件任务描述符 ZinAneTdHw_v10，一个七组寄存器映像，以 44 字节记录的稀疏序列化形式存储，其缓冲区基址是在加载时解析的重定位槽。

规范的程序形状是一个融合计算操作，前后由输入和输出格式转换包围。融合为一次引擎调度的网络解码为 Op0_Cast 后跟 Op1_AneInference 再后跟 Op2_Cast。

转换操作在输入时将主机半精度布局转换为引擎内部交错布局，在输出时转换回来；单个推理操作就是整个融合图。

每个操作有两个部分：一个参数帧，命名操作数绑定和每轴张量布局；以及一个属性 blob，保存内核参数。属性 blob 是一个嵌套的 FlatBuffer 子描述符，而非展开的微码：其字节多样性为 0.19，是结构化和稀疏记录的低值，描述符大小不随计算形状变化。内维度为 64 的收缩和内维度为 256 的收缩都产生相同的描述符大小，因此计算形状是描述符保存的一个参数，而非其展开的程序。

由此结构可得出两个性质。描述符是深度不变的：一个单操作图和一个六操作融合图都缩减为相同大小的单个推理操作。它跟踪的是调度次数而非操作次数：将图切割为三个段的桥接操作会产生三个推理操作和一个更大的描述符。

四个权重子流——偏置、后缩放、调色板查找和激活查找——各自拥有独立的重定位槽，因此带有折叠批归一化的卷积降级为四个独立的权重流。每个直接内存访问步幅是一个 26 位有符号字段，位于其寄存器字的位 6，针对每芯片边界表进行范围检查。张量的元素步幅变为这些寄存器字：每个引擎的行步幅、通道步幅、深度步幅和组步幅。

加载器修补的不仅仅是缓冲区基地址。每个重定位是一个自定义柱状（bar）操作，将解析后的值、设备地址、片上存储器数据集标识符或缓冲区偏移量写入由其寄存器、位偏移量和位宽度命名的精确描述符位字段中。运行时可变的形状和步幅作为实时输入参数分别绑定，每个参数与一个命名过程输入相关联，并针对起始、停止和步进进行范围检查，因此一个编译后的程序无需重新编译即可服务于一系列输入形状。

描述符布局按硅片代版本化：M1 构建 v10 映像，其他代在同一访问器族下构建不同版本，维度块、重定位寄存器地址和字段宽度在版本之间都发生变化。M1 的 v10 映像完全省略了计算缓存直接内存访问引擎：每个用于片上原子、计数器和等待事件原语的设置器都断言在此架构上不受支持。这就是为什么在 M1 上无法表达常驻原地状态缓冲区而退化为共享缓冲区的硅片级原因。任何读取任务描述符字节的工具都必须根据版本号进行分派，而不是假设固定布局。

序列化器稀疏地发出映像：只有当寄存器的值与架构默认值不同时，才会出现在流中，这就是为什么一个小型收缩只产生六到八个记录。每个存活的记录是固定的 44 字节：一个计数标记、一个 IOMMU aperture 中的寄存器地址，以及第二个 aperture 中写入 DMA 基地址寄存器的一个或两个设备地址。寄存器地址是每次加载的设备地址，由 IOMMU 虚拟化，因此同一逻辑寄存器在不同加载中解析为不同的地址。

23.4 解码后的程序

一个 64 到 64 的半精度线性层，使用单位权重矩阵，是 M1 主机上包含解析后的布局 sidecar 的最小真实程序。其容器是一个魔数为 0xbeefface 的 Mach-O 形状文件，带有一个伪架构标记和可执行文件类型。解析后的帧运行 [1, 1, 1, 64] 半精度输入到 [1, 1, 1, 64] 半精度输出，所有步幅均为 128 字节。

权重库以 512 字节步幅保存 16 个子内核。解码这些字节证明了该模型：8192 字节中只有 64 个非零，每个都是等于 1.0 的半精度值 0x3c00，分布在 16 个子内核中每个分区一个，即 64×64 单位矩阵的对角线。该容器是单位线性层的完整编译形式，通过乘累加阵列复制其输入。

调度描述符和容器一次对应一个网络。一个具有一个输入和五个保持状态的有状态循环模型解码为六个输入转换、一个融合推理操作和六个输出转换，状态缓冲区以 16 千字节步幅布局。整个融合图仍然是一个推理操作和一次调度。

23.5 磁盘上容器，逐段解码

第二个 M1 样本，一个 64 KiB 的 h13g 容器，确认了相同的格式并提供了以下差异；其段映射、+0x1c 的下一条记录链、0xC0 和 0x40 权重步幅、构建信息横幅和元素类型目录均与上述单位层解码匹配。证明该格式是 Mach-O 变体而非真正 Mach-O 的差异在于头部魔数：0xbeefface 是小端字节 CE FA EF BE，而真正的 64 位 Mach-O 会保存 0xfeedfacf。修补这四个字节后，otool 和 rabin2 可以解析该文件，而内核内加载器直接接受引擎魔数。

段命令命名了 __FVMLIB、__TEXT 和 __KERN_0 区域。两个 __FVMLIB 窗口声明了输入和输出张量，fileoff 和 filesize 均为零：它们没有磁盘上字节，仅作为虚拟内存声明存在，内核在提交时将其映射到地址转换 aperture 中。窗口段对齐到 0x4000（地址转换单元的 16 KiB 页粒度），而任务描述符和权重段对齐到 0x40（引擎的 64 字节 tile 粒度）。一对窗口绑定加载命令，每个是一个 {u32 size, u32 pad, u64 vmaddr, char name[8]} 记录，将符号名称 t0_ane 和 t5_ane 绑定到输入和输出虚拟地址。

23.5.1 任务描述符链表

该样本包含三个 ZinAneTd 记录，通过 +0x1c 处的下一个偏移量字段链接：0x000 到 0x300 到 0x500 再到终止符。每个描述符头部解码为表 23.11 给出的少量字段，包括链接列表的下一个偏移量字段。

表 23.11. ZinAneTd 任务描述符的解码头部字段，+0x1c 处为链接列表的下一个偏移量字段。

偏移量 字段 含义
+0x00 索引和标志 一个 16 位索引加上一个标志高字节，其中 0x03 标记最后一条或障碍记录
+0x04 大小 每个描述符的大小或延迟字段
+0x08 操作和引擎配置 一个操作类型字，其低字节区分引擎通道

每个描述符消耗的权重基地址不在头部中，而是在针对 __KERN_0 的内联重定位数组中。__text 上的 12 个重定位是 tile 基偏移量：第一个加权描述符读取 8 个 tile，第三个读取 4 个，中间的描述符没有内核重定位，因此是一个非加权通道，如激活、重塑或池化。

23.5.2 权重分块和每通道命名

__kern_0 节保存半精度权重系数，分为 64 字节对齐的 tile，每个引擎计算通道一个 tile，在符号表中命名为 K<sha256>_ne_<i>，其中索引 i 选择通道。该样本中的第一个权重 K596A4B73… 以 3×64 的 0xc0 字节卷积步幅分为八个 tile _ne_0 到 _ne_7。第二个权重 KE125552B… 以 0x40 字节矩阵乘法步幅分为四个 tile _ne_0 到 _ne_3，每个 tile 是一个小权重，填充到 64 字节 tile 粒度。符号表还列出了该第二个权重的 tile _ne_4 到 _ne_15，都指向节末尾作为未使用通道的空哨兵 tile，通过 desc 字段为零与活动 tile 上的 0x0002 区分。此样本展示的分块规则是：权重分为 min(8, 通道数) 个 tile，每个填充到 64 字节的倍数并连续布局，每个描述符通过重定位引用 tile i 的 64 字节对齐偏移量。

23.5.3 Stabs 风格形状描述符

输入和输出张量通过符号表中 stabs 风格的类型描述符来定义形状，这些条目的符号类型为 0x20，编码为 s<stride><axis> 字节步幅列表，涵盖四个轴：n 代表批次（batch），c 代表通道（channel），h 代表高度（height），w 代表宽度（width），如清单 23.6 中绑定的输入和输出所示。

清单 23.6. 绑定输入和输出张量的 stabs 风格形状描述符，编码了每轴字节步幅。

t0_ane (输入, vm 0x30000000): s8192n / s1024c / s64h / s2w
10826: t5_ane (输出, vm 0x30004000): s64n / s64c / s64h / s2w

输入步幅描述了一个八通道张量，按通道主序打包到 64 字节 tile 和 16 KiB 页分块中。2 字节的宽度步幅是一个半精度元素，64 字节的高度步幅是一个 tile 行，1024 字节的通道步幅是 16 个高度行。同一符号表枚举了完整的硬件元素类型目录，即在下面元素类型枚举部分解码的更宽集合。
